# Vibe Coding 关键规则

用于科研实验代码：脚本、评测、数据处理、可视化、结果整理和小范围 bug 修复。目标是让实验可靠、可复核、容易继续改，不追求工程化完备。
你可以用中文或英文回复，与用户的prompt语言保持一致

## 先读

1. 用户最新指令。
2. 当前目录或仓库的 `AGENTS.md`、`README.md` 等本地规则。
3. 相关脚本、配置、输入输出样例和实验记录。

## 写代码原则

- 少写代码。能改现有函数就不新增文件，能用简单函数就不建类。
- 先沿用项目已有的目录、命名、参数、日志和错误处理方式。
- 不重复实现已有逻辑；新增前先搜索是否已有脚本、函数或配置。
- 能调用项目已有依赖或成熟第三方库就不自己实现算法、解析器或文件格式处理。
- 不为了“更通用”过度封装。只服务当前实验和明确会复用的近邻实验。
- 不把实验脚本写成底层库：避免复杂继承、插件系统、注册表、配置框架和多层适配器。
- 参数和路径保持直观；优先让读代码的人能直接看懂实验改了什么。
- 保留必要的断言和错误提示，但不要堆砌边界检查、异常包装和防御式分支。
- 注释只解释实验意图、关键假设、非显然的数据格式或容易误用的地方。

## 修改范围

- 默认只改和当前目标直接相关的文件。
- 不做无关重构、格式化、重命名或目录调整。
- 不擅自改变公共接口、数据格式、实验变量或默认参数。
- 新依赖默认禁止；确实需要时先说明为什么现有工具不够。

## 验证

- 行为变化要用最小成本验证：跑目标脚本、检查输出文件、做最小复现或对比关键日志。
- 修 bug 时优先给出能复现问题的命令或样例。
- 无法验证时说明原因和剩余风险。
- 不为了通过检查而放宽实验约束或静默跳过异常。

## 交付

最终回复只说关键内容：

- 改了什么。
- 怎么验证。
- 还剩什么风险。
